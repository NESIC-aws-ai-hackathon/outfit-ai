# Requirements Document: 「今日の人格、着せておきました。」

## Intent Analysis

- **User Request**: 天気・予定・TPO・手持ち服から、AIが"今日のあなた"を服として決めるサービスの開発
- **Request Type**: New Project (Greenfield)
- **Scope Estimate**: System-wide — フルスタックWebアプリケーション（フロントエンド、バックエンド、AI連携、外部API連携）
- **Complexity Estimate**: Complex — AI統合、外部API連携、ユーザーデータ管理、画像処理、複数機能
- **Timeline**: 2〜3日（ハッカソン）

---

## 1. プロジェクト概要

### 1.1 サービスコンセプト
「今日の人格、着せておきました。」は、毎朝の服選びをAIに任せることで、その日の予定や天気に合った服装を自動で提案するWebサービスである。

AIは単なるコーデ提案ではなく、予定・天気・TPO・見られたい印象から「今日の人格」を定義し、それを手持ち服の中から服装として出力する。

### 1.2 ターゲットユーザー
- 朝の支度に時間をかけたくない人
- 服はあるのに組み合わせに悩む人
- TPOに合っているか不安になりやすい人
- 天気や気温に合わせた服選びが面倒な人
- 予定が複数ある日に服を決めるのが苦手な人

### 1.3 ハッカソンテーマ適合性：「人をダメにする」

本サービスは、朝の服選び・TPO判断・自己演出までAIに任せることで、ユーザーが自分で"今日の自分"を決めなくなる体験を提供する。AIは予定・天気・手持ち服をもとに"今日どう見られるべきか"を定義し、それを服装として出力する。

ここでいう「今日の人格」とは性格診断ではなく、予定・TPO・見られたい印象から定義される**外向きの印象設計**を指す。

使うほど、ユーザーは以下の意思決定を自分で行わなくなる：

| 段階 | 放棄される意思決定 | 結果 |
|------|------------------|------|
| 1 | 服の組み合わせを考える | AIが手持ち服から自動で選ぶ |
| 2 | 天気を確認して服を変える | AIが自動で天気を反映する |
| 3 | 予定に合う服を判断する | AIがTPOを自動判定する |
| 4 | 今日どう見られたいか考える | AIが「今日の人格」を定義する |
| 5 | 着たものを覚えておく | AIが着用履歴を管理する |

> **「使うほど、自分で今日の自分を決めなくなる」** — これがテーマ適合の核心である。

## 2. ユーザーストーリー（MVP）

以下は、MVPで実装する機能に紐づくユーザーストーリーである。各ストーリーは「便利さ」ではなく「意思決定の放棄」を価値として表現している。

### US-001: 手持ち服登録
- **As a** 朝に服を選ぶのが面倒なユーザー,
- **I want** AIがタグ付け・分類してくれる前提で手持ち服を登録しておきたい,
- **So that** コーデを考える材料すらAI管理になり、自分で組み合わせを考える余地がなくなる。
- **対応機能要件**: FR-002

### US-002: 予定の自動取得
- **As a** 予定に合う服を自分で考えたくないユーザー,
- **I want** 今日の予定がGoogle Calendarから自動で取り込まれてほしい,
- **So that** 予定の確認すら不要になり、AIが文脈を勝手に理解してくれる。
- **対応機能要件**: FR-003

### US-003: 天気情報の自動取得
- **As a** 天気を調べて服を変えるのが面倒なユーザー,
- **I want** 天気情報が自動で取得され服装判断に使われてほしい,
- **So that** 天気予報を見る判断すらAIに丸投げできる。
- **対応機能要件**: FR-004

### US-004: 今日の人格定義
- **As a** 自分で"今日の自分"を決めたくないユーザー,
- **I want** AIが予定・天気・TPOから「今日の人格」を定義してほしい,
- **So that** 自己演出・印象設計の意思決定を完全にAIに委ねられる。
- **対応機能要件**: FR-005

### US-005: AI服装提案
- **As a** 朝クローゼットの前で何も考えたくないユーザー,
- **I want** AIが「今日の人格」に基づいた具体的な服装を提案してほしい,
- **So that** 朝起きたら着るものが決まっている状態で1日を始められる。
- **対応機能要件**: FR-005

### US-006: 選定理由・除外理由の表示
- **As a** なぜその服が選ばれ、他の服が外されたのか気にはなるが自分では考えたくないユーザー,
- **I want** AIがその服を選んだ理由と、他の服を除外した理由を確認できること,
- **So that** 自分で判断を下さなくても「これでいい」と安心して委ねられる。
- **対応機能要件**: FR-005

### US-007: 着用履歴管理
- **As a** 同じ服ばかり着ていないか気にしたくないユーザー,
- **I want** AIが着用履歴を管理し連続着用を自動で避けてほしい,
- **So that** 着たものの記憶すらAI任せにでき、自分で覚えておく必要がなくなる。
- **対応機能要件**: FR-006

### US-008: 再生成
- **As a** 提案が気に入らないが自分で考え直したくないユーザー,
- **I want** 「別の提案」ボタン一つで再提案してほしい,
- **So that** 自分でコーデを組み直す労力なく、もう一度AIに丸投げできる。
- **対応機能要件**: FR-005

### US-009: ユーザー認証
- **As a** 自分の手持ち服・予定情報・着用履歴を安全に管理したいユーザー,
- **I want** アカウント登録・ログインにより個人単位でデータを保持したい,
- **So that** 一度登録すれば何も準備せずAIに委ねるだけの状態が維持される。
- **対応機能要件**: FR-001

## 3. 機能要件 (Functional Requirements)

### FR-001: ユーザー認証
- **説明**: メールアドレス＋パスワードによるユーザー登録・ログイン機能
- **優先度**: 高（MVP必須）
- **詳細**:
  - ユーザー登録（メール、パスワード、表示名）
  - ログイン / ログアウト
  - パスワードリセット機能
  - セッション管理

### FR-002: クローゼット管理（Closet Data）
- **説明**: ユーザーの手持ち服を登録・管理する機能
- **優先度**: 高（MVP必須）
- **登録項目**:
  - アイテム名
  - 種別（トップス、ボトムス、アウター、靴、アクセサリー等）
  - 色
  - 季節（春/夏/秋/冬/オールシーズン）
  - フォーマル度（1-5段階）
  - 使用可否（洗濯中等）
  - 最近着た日
  - 写真（画像アップロード）
  - お気に入り度
- **登録方法**: 手動入力を基本＋写真アップロード時のAIタグ補助
  - 写真をアップロードすると、AIが種別・色・フォーマル度を自動推定し、候補として表示
  - ユーザーが確認・修正して登録

### FR-003: 日次コンテキスト入力（Daily Context Input）
- **説明**: 今日の予定や条件を入力する機能
- **優先度**: 高（MVP必須）
- **入力項目**:
  - 今日の予定（時間帯ごと: 午前/午後/夜）
  - 移動手段・移動量
  - 見られたい印象（自由テキスト or 選択肢）
- **Google Calendar連携**:
  - Google Calendar APIと連携し、当日の予定を自動取得
  - 取得した予定をもとに、予定内容を自動入力
  - 手動での追加・編集も可能

### FR-004: 天気情報取得（Weather Context）
- **説明**: 天気APIから服装判断に必要な情報を自動取得
- **優先度**: 高（MVP必須）
- **取得項目**:
  - 最高気温 / 最低気温
  - 天気（晴れ/曇り/雨/雪等）
  - 降水確率
  - 湿度
  - 風速
  - 時間帯ごとの寒暖差
- **推奨API**: OpenWeatherMap API（無料枠: 60回/分、豊富なデータ、日本対応）

### FR-005: AI服装提案（Outfit Recommendation）
- **説明**: AIが天気・予定・TPO・手持ち服・着用履歴をもとに、今日の服装を提案する
- **優先度**: 高（MVP必須）
- **AI基盤**: Amazon Bedrock（Claude）
- **入力データ**:
  - ユーザーの手持ち服データ
  - 当日の天気情報
  - 当日の予定・コンテキスト
  - 着用履歴（直前の着用記録）
  - ユーザーの「見られたい印象」
- **出力**:
  - 「今日の人格」の定義（例: 「信頼感はあるが、話しかけやすい人」）
  - 推奨服装の組み合わせ（トップス、ボトムス、アウター、靴等）
  - 選定理由の説明
  - 除外理由の説明
- **TPO判定ロジック**:
  - 顧客訪問 → きれいめ、清潔感
  - 社内会議 → きちんと感と動きやすさのバランス
  - 展示会 → 歩きやすさ、疲れにくさ重視
  - デート → 清潔感、柔らかさ、好印象
  - 友人との食事 → カジュアル、少し華やか
  - 在宅勤務 → 快適性重視

### FR-006: 着用履歴管理
- **説明**: 提案した服の着用記録を簡易的に保存し、連続着用を避ける
- **優先度**: 中（MVP簡易版）
- **詳細**:
  - AIが提案→ユーザーが採用した服装を記録
  - 直前の着用記録を保持し、連続着用を避ける
  - 完全な履歴分析は将来対応

## 4. 非機能要件 (Non-Functional Requirements)

### NFR-001: パフォーマンス
- AI服装提案のレスポンス: 10秒以内
- ページロード: 3秒以内
- 天気API取得: 2秒以内

### NFR-002: 可用性
- ハッカソンデモ用のため、高可用性は不要
- 単一リージョンでの稼働で十分

### NFR-003: スケーラビリティ
- MVP段階: 同時ユーザー数10程度を想定
- 将来的な拡張を考慮した設計は行うが、過度な最適化は不要

### NFR-004: ユーザビリティ
- レスポンシブWebデザイン（モバイルファースト）
- 朝の利用を想定 → シンプルで直感的なUI
- 服装提案は1画面で確認可能

### NFR-005: セキュリティ（基本レベル）
- パスワードのハッシュ化
- HTTPS通信
- APIキーの安全な管理（環境変数）
- ※セキュリティ拡張ルールは適用しない（ハッカソンプロトタイプ）

## 5. 技術スタック（推奨）

### 5.1 フロントエンド
- **React + Next.js**: 最も普及したフレームワーク、レスポンシブ対応、SSR/SSG対応
- **Tailwind CSS**: 高速なUI開発
- **理由**: 大規模エコシステム、ハッカソンでの迅速な開発に適合

### 5.2 バックエンド
- **Python + FastAPI**: 軽量・高速、AWS SDK（boto3）との親和性、Bedrock連携が容易
- **理由**: AI/ML系タスクとの相性、迅速なAPI開発、型安全なAPI定義

### 5.3 データベース
- **Amazon DynamoDB**: サーバーレス、AWS統合、スキーマレスで迅速な開発
- **理由**: ハッカソンの2-3日タイムラインでのセットアップ速度、AWSインフラとの一体性
- **代替案**: PostgreSQL（RDS）— データの関係性が強い場合

### 5.4 AI/ML
- **Amazon Bedrock (Claude)**: コーデ提案・画像解析タグ補助

### 5.5 外部API
- **OpenWeatherMap API**: 天気情報取得
- **Google Calendar API**: 予定自動取得

### 5.6 インフラ
- **AWS**: デプロイ先
  - フロントエンド: Amazon S3 + CloudFront（または Amplify）
  - バックエンド: AWS Lambda + API Gateway（またはECS Fargate）
  - DB: DynamoDB
  - AI: Amazon Bedrock
  - 画像ストレージ: Amazon S3

## 6. MVP スコープ定義

### 6.1 MVP に含めるもの
- ユーザー登録・ログイン（メール+パスワード）
- クローゼット管理（手動入力＋写真アップロードAI補助）
- 日次コンテキスト入力（手動＋Google Calendar連携）
- 天気情報自動取得
- AI服装提案（「今日の人格」＋コーデ出力）
- 簡易着用履歴（直前の着用を記録、連続回避）

### 6.2 MVP に含めないもの（将来対応）
- 完全な着用履歴分析・統計
- ECサイト連携
- 複数カレンダーサービス対応
- 服の購入レコメンド
- ソーシャル機能（コーデ共有等）
- Outlook等のカレンダー連携
